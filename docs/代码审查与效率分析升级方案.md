# devops-ai 代码审查与效率分析 — 升级改造方案

## 一、改造总览

| # | 改造方向 | 当前痛点 | 目标效果 |
|---|---------|---------|---------|
| 1 | **OCR MCP 引入** | LLM 单次 prompt 调用，大 diff token 爆炸，行号漂移 | 接入 open-code-review MCP，逐文件行级精准审查 |
| 2 | **统一报告口径** | 三种口径混在一起，"78 个 commit"和"初始提交"出现在同一份报告 | 严格区分：①本次 diff 审查 ②分支范围审查 ③全量项目扫描 |
| 3 | **统一 Finding 数据结构** | OcrComment、CodeReviewResult、BugDetail 字段各自为政 | 15 个固定字段，所有问题统一出口 |
| 4 | **Bug 归因加状态（双 LLM 交叉验证）** | git blame 追到谁就直接定罪"引入了 bug" | 两个 LLM 各自给出置信度取平均，≥ 0.7 才确认 |
| 5 | **完整证据链** | 看到"第 42 行有空指针风险"，不知道怎么复现 | P0/P1 问题补齐：触发条件 + 修复建议 + 置信度 |
| 6 | **敏感信息脱敏** | 密码/token 原文展示在报告里 | 统一输出"疑似明文凭据，已命中 secret 规则"，值用 *** 代替 |
| 7 | **审查流程改造** | 大 diff 一次性灌给 AI | 先文件筛选 → 按业务链路分组 → 每组独立上下文调 AI |
| 8 | **二次过滤** | AI 输出第 500 行但文件只有 200 行——没人校验 | 第一轮找问题，第二轮校验行号/去重/误报/触发条件 |
| 9 | **报告结构调整** | 打开几十页，管理层找不到关键信息 | 首页只放管理摘要：风险等级 + P0/P1 + 待处理项 |
| 10 | **低贡献率开发者识别** | 只提交过 1-2 次的人在总览表里一行带过 | 占比低于平均占比的 25% 时自动标记 ⚠ + 展开全部 commit 明细 |
| 11 | **开发者贡献总结** | 只列 commit 清单，没人知道贡献画像 | 每人输出：类型分布、时间线、主要领域 |
| 12 | **开发者含 Bug 的 commit 识别** | 知道谁修了 bug，不知道谁引入的 | git blame 反向溯源 + 去重聚合 |
| 13 | **贡献者分析克制表述** | 报告直接推断"质量意识高/低" | 只陈述事实，标注"blame 关联 ≠ bug 引入"，不做人格推断 |
| 14 | **未修复漏洞的 blame 追溯** | 审查只能发现 bug，不知道是谁引入的 | 对审查发现的 P0/P1 漏洞做 git blame，追溯引入者 |

## 二、各改造方向详述

### 2.1 OCR MCP 引入提升代码审查能力

**当前问题：** `CodeReviewAiService` 把整个 diff 一次性塞进一个 LLM prompt，大变更集 token 爆炸，LLM 走捷径跳过文件；行号靠 LLM 自由描述，经常漂移；输出格式靠正则匹配 "### " 分隔符，格式一偏就解析失败。

**改造方案：** 保留 code-review-graph 架构分析能力不变，用 open-code-review MCP server 替换 LLM 直接调用。

```
CodeReviewAiService
  │
  ├─ OcrmcpClient（stdio JSON-RPC 通信）
  │    ├─ 启动 ocr serve 子进程
  │    ├─ initialize → tools/call
  │    ├─ code_review_diff：只审查变更行（有 git hash 时）
  │    └─ code_scan：全文件扫描（无 diff 上下文时）
  │
  └─ OCR 不可用时自动 fallback 旧 LLM 路径（保留兼容）
```

**ocr serve 内部：**
- 逐文件并发（goroutine 池）
- file_read → 读取完整文件
- code_search → 搜索符号/调用链
- code_comment → 结构化评论（path + line + existing_code + suggestion）
- 返回 JSON: `{status, comments[...], summary, warnings}`

**新增模型类：**

| 类 | 字段 |
|----|------|
| `OcrComment` | path, content, existingCode, suggestionCode, startLine, endLine, thinking |
| `OcrReviewResponse` | status, comments[], summary{filesReviewed, totalTokens, elapsed}, warnings[] |
| `OcrmcpClient` | 子进程管理 + JSON-RPC + 30min 超时 + isAvailable() 预检 |

> **注意：** OcrComment 是 OCR MCP 的**中间模型**，仅用于反序列化 OCR 返回的原始 JSON。下游全部通过 `Finding.fromOcrComment()` 转换为统一的 Finding 再流通。CodeReviewResult、BugDetail 同样会逐步收敛到 Finding，参见 2.3 节。

**效果：**
- 从"段落式模糊建议"升级为"行级精准评论 + diff 格式修改建议"
- 每行问题有确定性的 startLine/endLine，零漂移
- 自动统计 token 消耗和审查耗时

### 2.2 统一报告口径

**当前问题：** 当用户选了分支范围审查（比如从某个 commit 开始有 78 个 commit 迭代），生成的报告里前面写"78 个 commit 迭代"，后面又出现"初始提交/创建项目骨架"——因为全量扫描时自动把 root commit 当起点、分支审查时又用用户指定的范围，两种逻辑混在同一个 `generateSync()` 里，读者完全不知道这报告审的是什么范围。

**改造方案：** 新增 `ReviewScope` 枚举，**必须出现在每份报告的最前面**。

```java
enum ReviewScope {
    DIFF_REVIEW   ("本次 diff 审查", "仅检查两个 commit 之间的代码变更"),
    BRANCH_REVIEW ("分支范围审查",   "检查该分支上指定范围内的所有变更"),
    FULL_SCAN     ("全量项目扫描",   "扫描整个项目的所有源文件"),
}
```

**判断逻辑（在 GenerationOrchestrator 中）：**

| 条件 | 口径 |
|------|------|
| 用户指定了 sinceHash + untilHash，且 commits = 1 | ① 本次 diff 审查 |
| 用户指定了 sinceHash + untilHash，且 commits > 1 | ② 分支范围审查 |
| 无指定 hash，走全量扫描 | ③ 全量项目扫描 |

**报告中必须体现：**

> **审查范围**：分支范围审查（自 2025-01-15 起共 78 个 commit）

### 2.3 统一 Finding 数据结构

**当前问题：** 三个子系统各自定义自己的"问题"模型——OCR 输出用 `OcrComment`（path/content/existingCode/suggestionCode），审查结果用 `CodeReviewResult`（codeIssues/keyFindings/riskLevel），Bug 归因用 `BugDetail`（commitId/introducedBy/fixedBy）。字段不统一，互相转换时有信息丢失。

**改造方案：** 所有代码审查问题（不管来源是 OCR、AI 还是 Secret 检测）统一收敛到 `Finding`。OcrComment、CodeReviewResult、BugDetail 不再作为下游流通模型，仅在各子系统内部作为中间格式使用，最终都通过转换方法汇聚到 Finding。

```
数据流示意：

OCR MCP 输出 ──→ OcrComment ──→ Finding.fromOcrComment() ──┐
旧 LLM 路径   ──→ CodeReviewResult ──→ Finding.fromReview() ──┤
Bug 归因      ──→ BugDetail ──→ Finding.fromBugDetail() ────┤
Secret 检测   ─────────────────→ Finding 直接生成 ───────────┤
                                                              ↓
                                                    统一的 Finding 列表
                                                              ↓
                                              报告生成 / 统计 / 校验
```

**下游文件统一引用：** ReviewReportGenerator、EfficiencyReportGenerator、MarkdownGenerator 等报告和统计模块，全部从 Finding 取数据，不再直接依赖 OcrComment、CodeReviewResult 或 BugDetail。

```
Finding {
    id               // UUID 短码，唯一标识
    file             // 文件路径
    startLine        // 起始行号（1-indexed）
    endLine          // 结束行号（1-indexed）
    severity         // P0(阻断)/P1(高危)/P2(中危)/P3(低危)/P4(信息)
    category         // 空指针/安全漏洞/事务边界/并发安全/敏感信息/...

    // 置信度
    confidence            // 最终置信度 0.0-1.0，计算规则见下方"置信度计算规则"
    reviewConclusion     // 第二个 LLM 的复核结论，如"findById 返回 null 后直接调用 getName，确实存在 NPE 风险"

    // 证据链
    evidence         // 问题所在的代码片段
    trigger          // 什么输入/场景触发此问题
    suggestedFix     // 建议修复方式

    // 归属
    owner            // 责任人（来自 git blame）
    ownerEmail       // 责任人邮箱
    blameCommitIds   // 追溯到的 commit id 列表（同一段代码可能由多人在不同 commit 中编写）

    // 状态与复核
    status               // FindingStatus 枚举：CONFIRMED / FALSE_POSITIVE
}
```

> **置信度计算规则：**
> - 第一阶段：第一个 LLM（审查/分类 LLM）给出初始 `confidence`（如 0.8）
> - 第二阶段：第二个 LLM（review LLM）复核时给出复核置信度（如 0.6），系统取平均值更新 `confidence = (0.8 + 0.6) / 2 = 0.7`
> - 未复核时 `confidence` 保持第一个 LLM 的初始值不变
> - **展示规则**：`confidence ≥ 0.7` 展示为"已确认"，进入正式统计；`confidence < 0.7` 标记为"误报"，不展示

**辅助枚举：**

| 枚举 | 值 |
|------|----|
| `FindingSeverity` | BLOCKER(P0), HIGH(P1), MEDIUM(P2), LOW(P3), INFO(P4) |
| `FindingCategory` | SECURITY, NPE, TRANSACTION, CONCURRENCY, RESOURCE_LEAK, ERROR_HANDLING, SECRET_EXPOSURE, CODE_STYLE, PERFORMANCE, DEPENDENCY, ARCHITECTURE, LOGIC_ERROR, HARDCODED, OTHER |
| `FindingStatus` | CONFIRMED, FALSE_POSITIVE |

`OcrComment` 通过 `Finding.fromOcrComment()` 转换，保留下游兼容性。

### 2.4 Bug 归因加状态字段（双 LLM 交叉验证）

**当前问题：** fix commit + git blame 只能说明"修复的这几行代码是张三某次提交写的"，不能直接等于"张三引入了这个 bug"。现在的代码直接把 blame 结果等同于 bug 引入，统计中不做区分。

**改造方案：** 引入双 LLM 交叉验证机制——blame 追溯到原始 commit 后，不直接判定为 bug，而是让**另一个 LLM（review LLM）**去复核这个追溯结果。两个 LLM 各自给出置信度（0.0-1.0），取平均值作为最终置信度：≥ 0.7 展示为"已确认"，< 0.7 判定为"误报"不展示。

**验证流程：**

```
第一个 LLM（分类器）
  └─ 判断 commit 类别为 "Bug修复"
       │
       └─ git blame 追溯 → 找到引入的 commit 和作者
            │
            └─ 两个 LLM 各自独立判定：
                 ├─ 分类 LLM 给出的置信度：0.0-1.0（首次分类时产出）
                 ├─ review LLM 给出的置信度：0.0-1.0（独立复核时产出）
                 └─ 最终 confidence = (分类 LLM 置信度 + review LLM 置信度) / 2
                      │
                      ├─ confidence ≥ 0.7 → "已确认" → 进正式统计
                      └─ confidence < 0.7 → "误报" → 不进统计
```

**和单 LLM 直接判定的区别：**

| | 单 LLM 直接判定 | 双 LLM 交叉验证 |
|---|---|---|
| 判断方式 | 一个 LLM 说了算 | 两个 LLM 独立打分，取平均值 |
| 准确率 | 可能误判（把重构当 bug） | 两个 LLM 交叉验证，误判率大幅降低 |
| 核心逻辑 | 相信 AI 单次输出 | 置信度平均后 > 0.7 才认定，防止 AI 幻觉 |

**状态流转：**

```
分类 LLM 判定 "Bug修复" + git blame 追溯成功
  │
  ├─ review LLM 给出置信度，两个 LLM 置信度取平均
  │
  ├─ 平均 confidence ≥ 0.7 → "已确认" → 进正式统计
  └─ 平均 confidence < 0.7 → "误报" → 不进统计
```

**Review LLM 的 prompt 设计要点：**

```
你是代码审查专家。请判断以下 git blame 追溯结果是否确实代表一个 bug，
并给出你的置信度（0.0-1.0）。

Fix commit 信息：
  - 提交信息: "修复登录超时问题"
  - 修改了哪些行: [diff 内容]

Blame 追溯到的原始 commit：
  - 提交信息: "重构登录模块"
  - 作者: 张三

请判断：
  1. 原始代码是否确实存在缺陷（而不仅仅是被重构/优化/改需求）？
  2. 这个缺陷是否确实导致了后续修复所描述的问题？

注意区分：
  - 代码重构（行为不变但结构改变）≠ bug
  - 需求变更（原来逻辑对新需求不适用）≠ bug

输出格式（JSON）：
{"confidence": 0.8, "reason": "findById 返回 null 后未做检查直接调用 getName，确实存在 NPE 风险"}
```

**模型变更：** `BugDetail` 新增字段（BugDetail 后续收敛到 Finding，这些字段直接对应 `Finding.status` / `reviewConclusion`，不再重复定义）：

```java
// DeveloperEfficiency.java BugDetail 内部类新增（过渡方案，最终迁移到 Finding）
private FindingStatus attributionStatus;  // CONFIRMED(confidence ≥ 0.7) / FALSE_POSITIVE(confidence < 0.7)
private String reviewConclusion;          // 第二个 LLM 的复核结论

public boolean isConfirmed() {
    return attributionStatus == FindingStatus.CONFIRMED;
}
```

> **收敛说明：** BugDetail 的状态字段与 FindingStatus 枚举共享同一套值。两个 LLM 各自给出置信度取平均，≥ 0.7 为 CONFIRMED，< 0.7 为 FALSE_POSITIVE。报告展示时通过枚举映射为中文（如 `CONFIRMED → "已确认"`）。

**Review LLM 配置：** 使用与分类 LLM 不同的模型配置（可从 `ai_config` 表读取单独的 provider/model），确保两个 LLM 的判断是独立的。如果只有一个 LLM 可用，也可以用同一个 LLM 但换一个 system prompt（独立会话，不共享上下文）。

**报告中的体现：**

```
### 开发者 Bug 详情

#### 张三（45 次提交，关联 3 个，已确认 2 个）

| # | 问题提交 | 置信度 | 状态 | 复核结论 |
|---|---------|--------|------|------|
| 1 | bbb222 | 0.80 | 已确认 | "findById 返回 null 后未做检查直接调用 getName，确实存在 NPE 风险" |
| 2 | ddd444 | 0.30 | 误报 | "原始代码已使用 Optional.orElseThrow，后续修复只是为了改异常消息文案" |
| 3 | fff666 | 0.90 | 已确认 | "锁粒度过大导致并发场景下数据不一致" |

> 仅"已确认"（两个 LLM 置信度平均 ≥ 0.7）的记录计入正式统计。
```

### 2.5 完整证据链

**当前问题：** 审查报告里写"第 42 行有空指针风险"，研发看了不知道怎么复现、为什么现在没问题、应该怎么改。

**改造方案：** 每条 P0/P1 问题在报告中展示完整的三段证据：

| 字段 | 说明 | 示例 |
|------|------|------|
| `evidence` | 问题所在的代码片段 | `userMapper.findById(id)` 返回值未做 null 检查 |
| `trigger` | 触发条件 | 当传入的 id 不存在于数据库时，findById 返回 null |
| `suggestedFix` | 修复建议（含代码） | 用 `Optional.ofNullable(user).orElseThrow(...)` 包装 |

**报告输出示例：**

````
### UserService.java

**[P1] 第 42-45 行** 空指针风险 | 置信度 85% | 已确认

> 证据：
> ```java
> User user = userMapper.findById(id);
> return user.getName();
> ```
>
> 触发条件：当传入的 id 在数据库中不存在时
>
> 建议修复：
> ```diff
> - User user = userMapper.findById(id);
> - return user.getName();
> + User user = userMapper.findById(id);
> + if (user == null) {
> +     throw new NotFoundException("用户不存在: " + id);
> + }
> + return user.getName();
> ```
````

### 2.6 敏感信息脱敏

**当前问题：** 如果源代码中包含 `spring.datasource.password=MyP@ssw0rd`，报告里原样展示密码。

**改造方案：** 新增 `SecretDetector`，在审查完成后、报告生成前，用 11 条正则规则对所有 Finding 的 evidence/suggestedFix 等文本字段做脱敏扫描，将命中规则的凭据值替换为 `***`。

| 规则名 | 匹配目标 |
|--------|---------|
| `jdbc_password` | JDBC URL 中的 `password=` 参数 |
| `spring_datasource_password` | `spring.datasource.password:` 明文 |
| `db_password_env` | `DB_PASSWORD=` 环境变量 |
| `nacos_password` | `nacos.password:` 配置 |
| `config_password` | `config.password:` 通用配置 |
| `token_exposure` | 长字符串 token 赋值 |
| `api_key` / `secret_key` / `access_key` | API 密钥变量 |
| `plaintext_password` | `password="xxx"` 通用写法 |
| `private_key` | `-----BEGIN PRIVATE KEY-----` |

命中后不展示原文，统一输出：

> **[P0] 第 15 行** 敏感信息泄露
>
> 疑似明文凭据，已命中 secret 规则（spring_datasource_password）
>
> 触发：配置文件包含明文数据库密码
> 建议：将凭据移至环境变量或 Nacos Config，通过 `${DB_PASSWORD}` 引用

**脱敏处理：** `Finding.sanitize()` 在进入报告前对所有文本字段做正则替换，将密码值替换为 `***`。

### 2.7 审查流程改造

**当前问题：** 一次把几十个文件的 diff 塞进一个 LLM prompt，大文件超过 5000 字符就直接丢弃，AI 也无法跨文件理解 Controller → Service → Mapper 的完整调用链。

**改造方案：** 三步流水线替换原来的直接调用。

```
Step 1: 文件预筛选
  ├─ 跳过 DELETE 的文件
  └─ 跳过非代码文件（图片、二进制等）

Step 2: 按业务链路分组
  ├─ 从文件路径提取业务领域名
  │    UserController.java    ─┐
  │    UserService.java       ─┤ → 链路 "user"
  │    UserMapper.xml         ─┘
  │    OrderController.java   ─┐
  │    OrderService.java      ─┤ → 链路 "order"
  │    OrderMapper.xml        ─┘
  │
  └─ 每组的 background 包含：
      - 该链路的跨文件调用关系
      - code-review-graph 的架构风险信号
      - 审查重点（空指针、事务、并发等）

Step 3: 逐组调用 OCR MCP
  ├─ 每组的 path 参数 = 该组文件列表（逗号分隔）
  ├─ background = 链路上下文信息
  └─ OCR 内部用 file_read / code_search 按需读取相关文件，不灌大 diff
```

**提取业务链路的逻辑：**

```
从路径的最后一个有意义目录名提取：
  src/main/java/com/example/user/UserController.java  → "user"
  src/main/java/com/example/user/UserService.java     → "user"
  src/main/java/com/example/order/OrderService.java   → "order"
  src/main/resources/mapper/UserMapper.xml             → "user"
```

跳过框架目录名：java, src, main, resources, controller, service, mapper, dao, model, entity, config, util, common。

### 2.8 二次过滤

**当前问题：** AI/OCR 的原始输出没有校验环节——报告的"第 500 行有问题"可能文件只有 200 行，同个空指针问题被三条不同的建议描述，或者 try-catch 已经保护了的代码被识别为"未处理异常"。

**改造方案：** 新增 `FindingVerifier`，第一轮结果必须通过四道校验才能进最终报告。

```
原始 Finding 列表
  │
  ├─ 校验 1: 行号准确性
  │    startLine <= 0? → 拒绝
  │    endLine < startLine? → 修正 endLine = startLine
  │
  ├─ 校验 2: 去重
  │    同文件 + 行号范围重叠 → 合并为一个，保留较严重的 severity
  │
  ├─ 校验 3: 误报检测
  │    evidence 中已包含 try-catch / @Valid / if-null-throw / Optional.orElse?
  │    且置信度 < 0.8 → 降级为 P3(LOW)，状态标记为"误报"
  │
  └─ 校验 4: 触发条件完整性
       P0/P1 问题缺少 trigger?
       → 降级为 P2(MEDIUM)，不进高危列表
```

### 2.9 报告结构调整 — 四页标准格式

**当前问题：** 报告把 update log、贡献者分析、代码审查、效率分析全部从上到下串在一起，一个文件几十页，管理层要翻到最后才看到"到底有多少高危问题"。

- 顶部固定数据局限性注释块

### 2.10 低贡献率开发者识别和提交详情查看

**当前问题：** 一个项目几十个开发者，有的人只提交了 1-2 次，在总览表里一行带过，管理者完全不知道他们做了什么。

**改造方案：** 在 `GenerationOrchestrator.computeContributorStats()` 中：

1. 计算每个开发者提交占比
2. 计算平均占比 = 100% ÷ 开发者人数，占比 < 平均占比 × 25% → `lowFrequency = true`，报告中用 ⚠ 标记
3. 对低频率开发者，展开列出其全部 commit（ID、时间、分类、信息）

**模型（ContributorStats）：**

| 字段 | 说明 |
|------|------|
| `authorName` | 开发者姓名 |
| `commitCount` | 提交总数 |
| `percentage` | 占比（精确到小数点后一位） |
| `lowFrequency` | 是否低频率（占比 < 平均占比 × 25%） |
| `commitDetails` | 仅低频率时有值，每条含 commitId/message/category/date |
| `categoryDistribution` | 每个分类的 commit 数量 |
| `commitFrequency` | 按日期统计的提交频率 |

**报告输出示例：**

```
### 贡献者总览
| 排名 | 贡献者 | 提交数 | 占比 | 主要领域 |
|------|--------|--------|------|----------|
| 1 | 张三 | 45 | 35.2% | 新功能、Bug修复 |
| 2 | 李四 | 38 | 29.7% | 代码重构、性能优化 |
| 3 | 王五 ⚠ | 3 | 2.3% | 文档更新 |

### 王五（3 次提交，2.3%）⚠

- 提交类型分布：文档更新 2 次，Bug修复 1 次
- 提交时间线：2025-03-15（1）/ 2025-06-02（2）

#### 王五提交记录
| 提交ID | 时间 | 分类 | 提交信息 |
|--------|------|------|----------|
| a1b2c3d4 | 2025-06-02 | Bug修复 | fix: 修复登录页验证码显示 |
| e5f6g7h8 | 2025-04-01 | 文档更新 | docs: 更新 API 文档 |
| i9j0k1l2 | 2025-03-15 | 文档更新 | docs: 新增部署指南 |
```

**设计考量：**
- 非低频率开发者不展开 commit 明细（信息量太大），但保留类型分布+时间线
- 低频率标记仅作为**信息提示**，不在报告中解读为"贡献不足"
- 阈值采用相对比例（平均占比 × 25%）而非固定值（如 10%）：
  - 10 人团队：平均占比 10%，低于 2.5% 才标记 → 避免一半人被标 ⚠
  - 20 人团队：平均占比 5%，低于 1.25% 才标记 → 避免几乎全员标记
  - 该比例可通过配置项 `contributor.low-frequency-threshold` 调整（默认 0.25）

### 2.12 开发者含 Bug 的 commit 识别与统计

**当前问题：** 提交记录只显示谁修了多少 bug，但不知道这些 bug 是谁引入的，无法做效率归因。

**改造方案：** 基于 AI 提交分类 + JGit blame 反向溯源。

**核心流程：**

```
AI 分类器识别出 "Bug修复" 类 commit
  │
  └─ 对每个 fix commit:
       1. JGit 获取 parent → fix 的 diff
       2. 在 parent 版本上对每个被修改的文件运行 git blame
       3. 对被修改的每一行，查出 blame 源作者和源 commit
       4. 按作者去重 → 多人均摊 1/N 份额
       5. 跨文件去重（同一 fix 多个文件 blame 到同一 bug commit 时合并）
       6. 按 commitId 分组——每个 bug commit 是一个独立条目
  │
  ├─ 【AI 验证】对每个 blame 出的 bug commit，调用第二个 LLM（review LLM）
  │    两个 LLM 各自给出置信度，取平均值
  │    → confidence ≥ 0.7 → 保留此 blame 记录，状态 = CONFIRMED，进入聚合
  │    → confidence < 0.7 → 丢弃此 blame 记录（误报，blame 找错了，该 bug 非此 commit 引入）
  │
  └─ 聚合为每个开发者的效率画像（仅 CONFIRMED 参与聚合）
```

**去重示例：**

```
fix commit #aaa111（"修复登录超时"）
  ├─ LoginService.java:42 → blame → #bbb222, 张三
  ├─ LoginService.java:58 → blame → #bbb222, 张三    ← 同文件合并
  └─ AuthUtil.java:15     → blame → #bbb222, 张三    ← 跨文件合并

→ 张三 share = 1.0（1 个 bug-introducing commit，1 个人）

fix commit #aaa222（"修复金额计算"）
  ├─ OrderService.java:30 → blame → #bbb333, 张三
  └─ PriceUtil.java:12    → blame → #ccc444, 李四

→ 张三 share = 0.5，李四 share = 0.5（2 个人均摊）
```

**聚合逻辑：**

```java
aggregateFromBugs():
  Phase 1: 统计每人总提交数
  Phase 2: 按 blame 结果分配 bug 份额（仅 confidence ≥ 0.7 的 CONFIRMED 记录参与）
           - 对每个 fix commit，经 git blame + AI 验证后，confidence ≥ 0.7 的 blame 记录保留
           - confidence < 0.7 的 blame 记录直接丢弃（误报，blame 追溯到的人不是 bug 引入者）
           - BugDetail 不按 fixCommitId 去重（不同 fix 修同一个 bug commit 各算独立事件）
           - FixDetail 按 fixCommitId 去重（一个 fix commit 只算一次修复）
  Phase 3: 计算 blameAssociationRatio = bugsConfirmed / totalCommits
           （变量名使用 blameAssociationRatio 而非 bugRate，遵循 2.13 克制表述原则）
```

**报告结构：**

### 2.13 贡献者分析克制表述

**当前问题：** 旧报告的"关键观察"部分直接从数据推断人格——"Bug 率最高，质量意识有待提高""修复最多，质量意识强"。低贡献率被暗示为"不够努力"，高 bug 率被等同于"水平差"。这些推断缺乏上下文（可能他在攻坚最难模块、可能 bug 是别人改坏他修了、可能他只参与了项目一小部分），且容易引发团队矛盾。

**改造方案：** 三条原则——

**1. 只陈述事实，不做人格推断**

| ❌ 旧表述 | ✅ 新表述 |
|----------|----------|
| "张三 bug 率最高（12.5%），质量意识有待提高" | "本次分析范围内，张三有 12.5% 的 commit 被后续修复关联（已确认 2 个，误报 1 个）" |
| "李四质量意识好，没有引入任何 bug" | "本次分析范围内，未发现李四的 commit 被后续修复关联" |
| "王五修复最多，质量意识强" | "王五本次提交了 8 个修复类 commit" |
| "赵六贡献率仅 2%，参与度低" | "赵六本次提交 3 次，占比 2.3%" |

**2. 数据加注释说明局限性**

```
> **注意**：
> - git blame 关联仅表示修复涉及的历史代码行由对应开发者编写，
>   不一定意味着该开发者直接引入了 bug。
> - 低贡献率仅反映本次统计范围内的提交数量，不代表开发者
>   在其他方面的投入（如代码审查、设计讨论、技术文档等）。
> - 只有经两个 LLM 交叉验证（分类 LLM + review LLM 置信度平均 ≥ 0.7）
>   的条目才标记为"已确认"。
```

**3. 低贡献率仅作信号，不做解读**

- 仅标记 ⚠，不在文字上追加"参与度低""投入不足"等评论
- 提供 commit 明细方便管理者了解具体做了什么，但不替管理者做判断

### 2.14 未修复漏洞的 blame 追溯

**为什么需要这个功能：**

当前项目已经做了两件关键的事——代码审查（通过 OCR/AI 发现系统漏洞）和 Bug 归因（对已修复的 bug 做 blame 追溯）。但这两件事之间是断开的：

```
代码审查 → 发现 10 个空指针风险 → 报告里列出了 → 然后就没了
Bug 归因 → 找到已修复的 bug → 追溯谁引入的 → 生成效率统计
```

审查发现的漏洞是谁引入的？没有人追溯过。

如果只追踪"已修复的 bug"而不管"还没修的漏洞"，效率分析就只能看到历史：谁写的代码被别人修过多少次。而代码审查发现的是**当下系统中真实存在的风险**——这些漏洞还没人修，它们才是真正可能引发事故的。追溯这些漏洞的引入者，比追溯已修复 bug 更有价值：

| 维度 | 已修复 bug 追溯 | 未修复漏洞追溯 |
|------|----------------|--------------|
| 关注点 | 过去发生的事 | 当前存在的风险 |
| 价值 | 事后复盘 | 事前预防 |
| 时效性 | bug 已经修了，信息偏历史 | 漏洞还没修，直接影响代码质量 |
| 可操作性 | 只能看谁"曾经"写了有问题的代码 | 可以推动责任人及时修复 |

**改造方案：** 在审查流水线中新增 `FindingBlameTracer` 和 review LLM 交叉验证——审查完成后，先对每条 P0/P1 的 Finding 执行 git blame 填充归属信息，再经 review LLM 独立判定是否为真实漏洞，最后通过 SecretDetector 脱敏和 FindingVerifier 校验，完整流程如下。

**完整流程：**

```
代码审查流水线
  │
  ├─ OCR/AI 审查 → List<Finding>
  │    每个 Finding 已有: file, startLine, endLine
  │
  ├─ 【新增】FindingBlameTracer
  │    对每个 P0/P1 的 Finding:
  │      git blame -L <startLine>,<endLine> <file>
  │      → 填充 owner, ownerEmail, blameCommitIds
  │      → 多人共写同一段代码时：按行数比例分配 blame 份额
  │
  ├─ review LLM 交叉验证（独立 LLM，与分类 LLM 不同会话）
  │    输入: Finding 的 evidence + blame 追溯结果（owner, blameCommitIds, commit message）
  │    两个 LLM 各自给出置信度，取平均值：
  │    → confidence ≥ 0.7 → Finding.status = CONFIRMED
  │    → confidence < 0.7 → Finding.status = FALSE_POSITIVE
  │    → 写入 Finding.reviewConclusion
  │    → 更新 Finding.confidence = (原 confidence + review 置信度) / 2
  │
  ├─ SecretDetector 脱敏
  │
  ├─ FindingVerifier 二次校验
  │
  └─ 生成报告
```

> **对未修复漏洞的 review LLM 验证，与 2.4 节已修复 bug 的双 LLM 验证逻辑一致，review LLM 使用独立的模型配置和会话上下文，确保交叉验证的有效性。**

```
