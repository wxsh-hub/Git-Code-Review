# Phase 8: 报告结构调整 + 克制表述

## 目标

1. 将报告重构为**四页标准格式**：管理摘要 → 问题处置页 → 模块与趋势页 → 效率与贡献附录
2. 补充问题处置所需的新字段（候选处理人、复核人、截止时间）
3. 新增模块与趋势页，按模块聚合风险、展示重复问题类型
4. 将效率分析中的主观推断改为克制的事实陈述，消除可能引发团队矛盾的表述

## 背景

**结构问题：** 当前报告把 update log、贡献者分析、代码审查、效率分析全部从上到下串在一起，一个文件几十页，管理层要翻到最后才看到"到底有多少高危问题"。

**表述问题：** 旧报告的"关键观察"部分直接从数据推断人格——"Bug 率最高，质量意识有待提高""修复最多，质量意识强"。低贡献率被暗示为"不够努力"，高 bug 率被等同于"水平差"。这些推断缺乏上下文，且容易引发团队矛盾。

**目标受众分层：** 不同角色关心不同的信息——高管看摘要和趋势，研发经理看处置清单，总监看模块风险分布，管理者复盘看效率附录。一份报告四种读法，每一页对应一个角色。

### 8.2 第一页：管理摘要

**位置：** 报告首页，所有内容控制在一页内。

**包含内容（以下六项）：**

#### 1) 审查范围

从 ReviewScope 获取，标注口径：

> **审查范围**：分支范围审查（自 2025-01-15 起共 78 个 commit）

#### 2) 版本/分支

```
| 项目 | xxx-api |
| 版本 | v2.3.1 |
| 分支 | feature/order-refactor |
| 审查时间 | 2025-06-15 14:30:00 |
```

#### 3) 风险等级

驱动自 P0/P1 计数：

| P0 数量 | P1 数量 | 风险等级 |
|---------|---------|---------|
| ≥ 3 | - | 🔴 严重 |
| 1-2 | - | 🟡 高 |
| 0 | ≥ 5 | 🟡 高 |
| 0 | 1-4 | 🔵 中 |
| 0 | 0 | 🟢 低 |

#### 4) 问题统计表

```
| 级别 | 数量 | 已确认 | 误报 |
|------|------|--------|------|
| P0 阻断 | 2 | 2 | 0 |
| P1 高危 | 5 | 3 | 2 |
| P2 中危 | 12 | 12 | 0 |
```

#### 5) 是否建议发布

基于 P0/P1 数量自动生成：

| 条件 | 建议 |
|------|------|
| P0 > 0 | ⛔ **不建议发布** — 存在 P0 阻断性问题，须全部修复后重新审查 |
| P0 = 0，P1 ≥ 3 | ⚠ **建议修复后发布** — P1 高危问题较多，建议修复后再上线 |
| P0 = 0，P1 1-2 | ⚠ **有条件发布** — 少量 P1 问题，建议排入下个迭代修复 |
| P0 = 0，P1 = 0 | ✅ **建议发布** — 无阻断/高危问题，可正常上线 |

#### 6) 需要谁在什么时候处理

> **待处理事项：**
>
> | 优先级 | 问题 | 文件 | 处理人 | 截止时间 |
> |--------|------|------|--------|---------|
> | ⛔ P0 | 敏感信息泄露 | application.yml | 张三 | 立即 |
> | ⛔ P0 | SQL 注入风险 | UserMapper.xml | 李四 | 立即 |
> | ⚠ P1 | 空指针风险 | UserService.java:42 | 张三 | 2025-06-18 |
> | ⚠ P1 | 事务边界缺失 | OrderService.java:88 | 王五 | 2025-06-20 |

**处理人指派规则：**
- 优先使用 blame 追溯到的 owner
- 无 blame 信息时使用文件最后修改者
- 都不存在时标注"待指派"

**截止时间生成规则：**
- P0 → "立即"
- P1 → 审查日期 + 3 个工作日
- P2 → 审查日期 + 7 个工作日

**重要原则：** 管理摘要不放任何推测性文字、不点名具体开发者（处理人栏位除外）、不做价值判断。

### 8.3 第二页：问题处置页

**受众：** 主管和研发经理，需要逐条了解问题并安排修复。

**每条问题的完整字段：**

| 字段 | 说明 | 来源 |
|------|------|------|
| **严重度+编号** | P0/P1/P2/P3/P4 + 序号 | Finding.severity |
| **代码链接** | 文件路径 + 行号（可点击跳转） | Finding.file + startLine/endLine |
| **问题标题** | 一行摘要描述 | Finding.category 映射 |
| **证据** | 问题代码片段（语法高亮） | Finding.evidence |
| **触发条件** | 什么输入/场景触发 | Finding.trigger |
| **建议修复** | diff 格式的修复建议 | Finding.suggestedFix |
| **候选处理人** | 推荐谁来修 | Finding.candidateHandler（blame 追溯） |
| **状态** | 已确认 / 误报 | Finding.status |
| **复核人** | 谁复核的 | Finding.reviewer（Phase 6 填充，"review LLM"） |
| **截止时间** | 建议修复截止日期 | 自动计算（见 8.2 §6） |

**排序规则：**
1. 先按严重度排序：P0 → P1 → P2 → P3 → P4
2. 同严重度内按文件路径排序
3. 同文件内按行号排序

### 8.4 第三页：模块与趋势页

**受众：** 经理和总监，关注风险分布和系统性问题。

**这是全新页面，当前系统没有对应实现。**

#### 8.4.1 按模块统计风险

```
## 模块风险分布

| 模块 | 文件数 | P0 | P1 | P2 | P3 | P4 | 风险分 |
|------|--------|----|----|----|----|----|--------|
| user | 12 | 1 | 2 | 3 | 1 | 2 | 25 |
| order | 8 | 1 | 1 | 4 | 0 | 1 | 20 |
| payment | 5 | 0 | 2 | 2 | 1 | 0 | 12 |
| product | 10 | 0 | 0 | 2 | 3 | 1 | 5 |
| common | 15 | 0 | 0 | 1 | 2 | 2 | 3 |
```

**风险分计算规则：**
- P0 = 10 分，P1 = 5 分，P2 = 2 分，P3 = 1 分，P4 = 0 分
- 模块风险分 = Σ(各问题 × 权重)
- 按风险分降序排列

**模块划分规则（与 Phase 4 的业务链路分组一致）：**
- 从文件路径提取业务领域名
- 跳过框架目录：java, src, main, resources, controller, service, mapper, dao, model, entity, config, util, common
- 无法归入业务模块的文件归入"其他"

### 8.6 克制表述改造

在 `EfficiencyReportGenerator.generateKeyObservations()` 中重写。

**三条原则：**

**1. 只陈述事实，不做人格推断**

| ❌ 旧表述 | ✅ 新表述 |
|----------|----------|
| "张三 bug 率最高（12.5%），质量意识有待提高" | "本次分析范围内，张三有 12.5% 的 commit 被后续修复关联（已确认 2 个，误报 1 个）" |
| "李四质量意识好，没有引入任何 bug" | "本次分析范围内，未发现李四的 commit 被后续修复关联" |
| "王五修复最多，质量意识强" | "王五本次提交了 8 个修复类 commit" |
| "赵六贡献率仅 2%，参与度低" | "赵六本次提交 3 次，占比 2.3%" |
| "代码质量整体一般" | "本次审查共发现 5 个 P0/P1 问题，分布在 3 个文件中" |
| "这个模块的风险较高" | "该模块发现 3 个 P0 问题，建议优先关注" |

**2. 数据加注释说明局限性**

每份报告的效率分析章节前，固定加入以下注释块：

```
> **注意**：
> - git blame 关联仅表示修复涉及的历史代码行由对应开发者编写，
>   不一定意味着该开发者直接引入了 bug。
> - 低贡献率仅反映本次统计范围内的提交数量，不代表开发者
>   在其他方面的投入（如代码审查、设计讨论、技术文档等）。
> - 只有经两个 LLM 交叉验证（分类 LLM + review LLM 置信度平均 ≥ 0.7）
>   的条目才标记为"已确认"。
> - 未修复漏洞的引入者信息同样来自 git blame，已通过 review LLM 交叉验证，
>   但最终判定仍需人工复核。
```

**3. 低贡献率仅作信号，不做解读**

- 仅标记 ⚠，不在文字上追加"参与度低""投入不足"等评论
- 提供 commit 明细方便管理者了解具体做了什么，但不替管理者做判断
- 不在"协作信号"章节提及低频率贡献者

### 8.7 "关键观察" → "协作信号"

重命名章节并重写内容：

```java
sb.append("### 协作信号\n\n");

// 只列事实，每条以数据驱动，不带主观形容词
// 示例：
- "张三 3 个 commit 被后续修复关联（0 个已确认）"
- "李四 本分析范围提交 8 个修复类 commit"
- "共 15 个 fix commit，对应 20 条 blame 关联记录"
- "经双 LLM 交叉验证：已确认 5 个，误报 15 个"
- "审查发现 7 个 P0/P1 未修复漏洞，涉及 3 个文件"

// 末尾加注释（同 8.5 的注释块）
sb.append("> **注意**：git blame 关联 ≠ bug 引入...\n");
```

**协作信号的生成规则：**
- 有已确认 bug 关联的开发者 → 列出数量和比例
- 修复了大量 bug 的开发者 → 列出修复数
- 有未修复漏洞关联的开发者 → 列出漏洞数（已确认/总数）
- 什么都不说的情况：以上都不满足的开发者
- **禁止生成的表述：** "质量意识"、"水平"、"能力"、"态度"、"努力"、"优秀"、"差"、"最好"、"最差"

### 8.8 Finding 模型字段确认

以下字段用于问题处置页和模块趋势页。所有字段已在 Phase 1 的 Finding.java 中定义，由各前置 Phase 填充值，Phase 8 仅消费：

```java
// 问题处置相关（Phase 8 使用，前置 Phase 已填充）
String codeLink;           // Phase 8 当场拼接：
                           //   gitRemoteUrl(Phase 4) + branch + filePath + #LstartLine-LendLine

**codeLink 生成规则：**

- 无法获取 git remote 时跳过

## 验收标准

**四页结构：**
- [ ] 报告第一页为管理摘要，内容控制在一页以内
- [ ] 管理摘要包含六项：审查范围、版本/分支、风险等级、问题统计、是否建议发布、处理人+截止时间
- [ ] 管理摘要不放推测性文字、不做价值判断
- [ ] 第二页为问题处置页，按严重度排序（P0 → P1 → P2 → P3 → P4）
- [ ] 每条问题包含全部 8 个字段：代码链接、证据、触发条件、建议修复、候选处理人、状态、复核人、截止时间
- [ ] P3/P4 折叠为一行摘要，不展开完整证据链
- [ ] 第三页为模块与趋势页，包含模块风险分布和重复问题类型
- [ ] 风险分计算正确（P0=10, P1=5, P2=2, P3=1, P4=0）
- [ ] 模块划分与 Phase 4 业务链路分组一致
- [ ] 第四页为效率与贡献附录，包含贡献者统计、blame 关联、修复记录、低频提交者明细
- [ ] 四页内容不重复、不交叉

**问题处置：**
- [ ] 代码链接可点击跳转到 GitLab/GitHub 对应行
- [ ] 候选处理人优先使用 blame owner，其次文件最后修改者，都不存在标注"待指派"
- [ ] P0 截止时间显示"立即"，P1 显示审查日期+3工作日，P2 显示审查日期+7工作日
- [ ] P0 问题前有 ⛔ 标记，P1 有 ⚠ 标记

**发布建议：**
- [ ] 发布建议判定逻辑正确（P0>0 → 不建议发布，P0=0 P1≥3 → 建议修复后发布，P0=0 P1 1-2 → 有条件发布，P0=0 P1=0 → 建议发布）

**模块趋势：**
- [ ] 重复问题类型正确聚合（同 FindingCategory ≥ 2 次才展示）
- [ ] 趋势信号仅在有历史数据时展示
- [ ] 模块风险分布按风险分降序排列

**克制表述：**
- [ ] "关键观察"章节改名为"协作信号"
- [ ] 所有人格推断（"质量意识高/低""水平好/差"等）不存在
- [ ] 低贡献率仅标记 ⚠，无附加评论
- [ ] 每份报告包含数据局限性注释块
- [ ] 变量命名使用 blameAssociationRatio 等克制名称
- [ ] 协作信号仅列数据事实，不推导人格
- [ ] 效率与贡献附录中没有"Bug 率最高""修复最多""产生 bug 最多"等人格化排名标题

## 依赖关系

Phase 8 是本方案的**最终阶段**，依赖前置所有阶段：
- Phase 1：Finding 统一数据结构（问题处置页的数据基础）
- Phase 4：ReviewScope + 业务链路分组（管理摘要审查范围 + 模块划分规则）
- Phase 5：FindingBlameTracer（问题处置页的候选处理人 + blame 信息）

- Phase 7：贡献者统计 + 效率分析（第四页附录的全部数据）
