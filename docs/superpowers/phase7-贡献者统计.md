# Phase 7: 贡献者统计（低频率 + 总结 + 效率分析）

## 目标

1. 低频率贡献者自动识别和标记，展开其全部 commit 明细
2. 每个开发者输出完整的个人贡献画像（类型分布、时间线、主要领域）
3. 基于 AI 提交分类 + git blame 反向溯源，产出开发者效率分析

## 背景

当前报告对贡献者的处理比较粗略：
- 只提交过 1-2 次的人在总览表里一行带过，管理者完全不知道他们做了什么
- 只列 commit 清单按分类汇总，看不到每个开发者的个人画像
- 只知道谁修了 bug，不知道这些 bug 是谁引入的

Phase 7 分两部分：**贡献者统计**（低频率识别 + 个人画像）独立于 Phase 5/6，可先行开发；**效率分析聚合**（Bug 识别与统计）依赖 Phase 6 的双 LLM 交叉验证结果（仅 CONFIRMED 记录参与聚合），需等待 Phase 6 完成后接入。Phase 4 产出的 ReviewScope 是全部功能的前置依赖。

> **数据来源说明：** 效率分析中"未修复漏洞详情"的引入者、置信度、状态字段来自 Phase 5（blame 追溯）+ Phase 6（双 LLM 交叉验证）。Phase 7 消费这些数据做聚合和渲染，不负责追溯或验证。

## 要做的事情

### 7.1 低频率贡献者识别

在 `GenerationOrchestrator.computeContributorStats()` 中实现：

**判定逻辑：**
1. 计算每个开发者提交占比 = 该开发者提交数 ÷ 总提交数 × 100%
2. 计算平均占比 = 100% ÷ 开发者人数
3. 占比 < 平均占比 × 25%（阈值）→ `lowFrequency = true`，报告中用 ⚠ 标记

**阈值设计考量：**
- 采用相对比例而非固定值
- 10 人团队：平均 10%，低于 2.5% 才标记 → 避免一半人被标 ⚠
- 20 人团队：平均 5%，低于 1.25% 才标记 → 避免几乎全员标记
- 阈值可通过配置项调整：

```yaml
contributor:
  low-frequency-threshold: 0.25  # 默认 0.25（平均占比的 25%）
```

### 7.2 修改 `ContributorStats.java`

路径：`devops-ai-core/src/main/java/com/devops/ai/core/model/ContributorStats.java`

新增和确认字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| authorName | String | 开发者姓名 |
| commitCount | int | 提交总数 |
| percentage | double | 占比（精确到小数点后一位） |
| lowFrequency | boolean | 是否低频率（占比 < 平均占比 × 25%） |
| commitDetails | List\<CommitDetail\> | 仅当 lowFrequency=true 时有值 |
| categoryDistribution | Map\<String, Integer\> | 每个分类的 commit 数量 |
| commitFrequency | Map\<String, Integer\> | 按日期统计的提交频率（按月份聚合） |

**CommitDetail 内嵌类：**

| 字段 | 类型 | 说明 |
|------|------|------|
| commitId | String | 提交 ID（短码） |
| message | String | 提交信息 |
| category | String | AI 分类结果 |
| date | LocalDateTime | 提交时间 |

**设计原则：**
- 非低频率开发者不展开 commit 明细（信息量太大），但保留类型分布+时间线
- 低频率标记仅作为**信息提示**，不在报告中解读为"贡献不足"
- 不替管理者做判断，只呈现数据

### 7.4 开发者含 Bug 的 commit 识别（效率分析）

基于 AI 提交分类 + JGit blame 反向溯源，产出每个开发者的效率画像。

**核心流程与聚合逻辑见 Phase 6 §6.5（`aggregateFromBugs()`）。** 本阶段的职责是以 Phase 6 产出的 CONFIRMED/BugDetail 数据为基础，按 §7.5 的格式渲染报告中的效率分析章节。要点回顾：

- Bug 识别与 blame 追溯 → Phase 6 的 `DeveloperEfficiencyService.aggregateFromBugs()` 负责
- 双 LLM 交叉验证 → Phase 6 负责，仅 `confidence ≥ 0.7` 的 CONFIRMED 记录参与聚合
- **本 Phase** 消费聚合后的数据，生成报告中的开发者 Bug 详情表和汇总排名

## 验收标准

- [ ] 低频率判定使用相对比例（平均占比 × 25%），阈值可通过配置项调整
- [ ] 低频率贡献者在总览表中显示 ⚠ 标记
- [ ] 低频率贡献者展示完整 commit 明细（ID、时间、分类、信息）
- [ ] 非低频率贡献者不展开 commit 明细但保留类型分布+时间线
- [ ] 每人详情卡片包含：类型分布、时间线、主要领域
- [ ] Bug 引入排名仅基于 CONFIRMED 记录
- [ ] 去重逻辑正确：跨文件合并、按 commitId 分组
- [ ] BugDetail 不按 fixCommitId 去重，FixDetail 按 fixCommitId 去重
- [ ] 份额计算正确：多人 blame 到不同 bug commit 时均摊
- [ ] 报告包含"Bug 引入排名"和"Bug 修复排名"两张汇总表
- [ ] 未修复漏洞详情独立章节，包含引入者、置信度、状态
- [ ] 开发者综合画像表同时展示已修复和未修复两个维度
