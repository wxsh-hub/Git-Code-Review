# 代码审查文档生成功能 — 设计文档（修正版）

## 1. 需求概述

在现有**发布文档（CHANGELOG）生成**的基础上，增加一个选项开关，勾选后额外基于**代码变更内容**生成**代码审查文档**。审查文档基于两阶段流水线：

1. **code-review-graph** — 不依赖 AI，纯静态分析：**获取实际源代码**，解析变更文件的 diff，构建**代码的依赖关系图、调用链、变更影响范围**，输出结构化图数据（JSON/DOT）
2. **AI 大模型分析** — 将图数据 + **代码变更 diff** 作为上下文，由 LLM 进行深度审查

> ⚠️ 关键区别：审查对象是**变更的源代码本身**（新增/修改/删除了哪些代码），而不是 commit message。commit message 仅作为辅助上下文。

## 2. 数据获取流程

```
从 GitLab 获取两阶段代码:
  1. 基于 sinceHash/untilHash（或时间范围）确定变更区间
  2. 获取变更涉及的 commit 列表
  3. 对每个变更文件，获取:
     a. 变更前版本（基准版本）的源码
     b. 变更后版本（目标版本）的源码
  4. 生成行级 diff（统一 diff 格式）
  5. 将变更代码送入 graph 引擎做静态分析
```

### 源码获取方式（二选一）

| 方式 | 适用场景 | 实现 |
|------|---------|------|
| **JGit Clone**（已有 `GitCloneService`） | 本地已有仓库或允许 clone | 直接 `git diff sinceHash..untilHash -- files` 获取 diff，`git show hash:path` 获取单个文件内容 |
| **GitLab API**（已有 `GitLabService`） | 无需本地仓库 | `GET /projects/:id/repository/compare?from=sha&to=sha` 获取 diff，`GET /projects/:id/repository/files/:path/raw?ref=sha` 获取单个文件内容 |

推荐优先使用 JGit Clone 方式，因为：
- 一次 clone 后所有操作本地完成，速度快
- `git diff` 直接生成结构化 diff
- 便于 graph 引擎对完整源码树做依赖分析

## 3. 与现有系统的关系

```
现有流程（不变）：
  项目配置 → 获取Commits(元数据) → 去重 → 分类 → 模板渲染 → 发布文档(MD/HTML)

新增流程（并行，可选）：
  项目配置 → 获取Commits(元数据)
                                  ↓ (可选 "生成代码审查文档")
                  获取源码 → 生成Diff → code-review-graph(静态分析)
                                             ↓
                                        AI 大模型(代码深度审查)
                                             ↓
                                       审查文档(MD/HTML)
```

审查流程和发布文档流程**共享 commit 元数据获取**，但审查流程额外需要**获取源码文件**和**生成 diff**，这是独立的耗时操作。

## 4. 两阶段流水线详细设计

### 4.1 第一阶段: code-review-graph（纯静态分析，不依赖 AI）

```
输入: 项目Git仓库 + sinceHash + untilHash
 ↓
┌─────────────────────────────────────────────────┐
│ Step 1: 获取变更文件列表及 Diff                   │
│   - git diff sinceHash..untilHash --name-only    │
│   - git diff sinceHash..untilHash (完整diff)     │
│   - 输出: 变更文件清单 + 每个文件的统一diff格式    │
└─────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────┐
│ Step 2: 对每个变更文件做源码解析                  │
│   - 解析 Java 文件: package, import, 类声明,      │
│     方法签名, 注解, 字段                          │
│   - 识别: 新增/修改/删除的方法、字段、注解         │
│   - 输出: 每个文件的 AST 节点变更清单             │
└─────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────┐
│ Step 3: 构建依赖关系图谱                          │
│   文件级依赖图:                                   │
│     - 节点 = 文件 (含包名)                        │
│     - 边 = import 关系                           │
│   类级依赖图:                                     │
│     - 节点 = 类/接口                              │
│     - 边 = 继承/实现/关联                         │
│   变更标注: 标记每个节点是 新增/修改/未变更/受影响  │
│  输出: CodeReviewGraph (JSON)                    │
└─────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────┐
│ Step 4: 影响范围分析 (ImpactAnalyzer)             │
│   - 从变更节点出发，沿依赖边反向遍历               │
│   - 找出所有可能受影响的文件/模块                  │
│   - 按影响程度分级: 直接影响/间接影响/可能影响     │
│   - 识别高风险信号:                               │
│     • 公共接口/API 变更                           │
│     • 核心模块（被大量依赖）被修改                  │
│     • 数据库实体/SQL 变更                         │
│     • 配置文件的修改                              │
│  输出: ImpactScope                                │
└─────────────────────────────────────────────────┘
 ↓
 输出: 结构化图数据 CodeReviewGraph + ImpactScope (JSON)
```

#### 代码解析范围

针对 Java 项目，graph 引擎需要识别以下元素：

| 元素 | 识别方式 | 审查价值 |
|------|---------|---------|
| package 声明 | 正则/行解析 | 模块归属 |
| import 语句 | 正则/行解析 | 文件级依赖 |
| 类/接口声明 | 正则扫描 class/interface/enum | 架构结构 |
| 方法签名 | 正则扫描访问修饰符+返回类型+方法名+参数 | API 变更检测 |
| 字段声明 | 正则扫描 | 数据模型变更 |
| 注解 | 正则扫描 @Annotation | 框架配置变更（如 @Transactional） |
| 泛型参数 | 正则扫描 | 类型约束变更 |

> 当前阶段使用正则 + 行解析即可满足需求，后续可用 JavaParser 做完整 AST 解析。

### 4.2 第二阶段: AI 大模型分析

```
输入: CodeReviewGraph + ImpactScope + FileDiff 列表
 ↓
┌─────────────────────────────────────────────────┐
│ Step 1: 组装 AI Prompt                           │
│   包含:                                          │
│   - 项目/版本元信息                              │
│   - 变更 commit 列表（标题，不含详细 message）    │
│   - 变更文件总览                                 │
│   - 每个变更文件的 diff（重点：新增/修改的代码）   │
│   - 依赖关系图谱摘要（JSON 转结构化文本）          │
│   - 影响范围分析结果                             │
└─────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────┐
│ Step 2: 调用 LLM 进行深度审查                     │
│   一次请求包含所有维度（节省 token）：             │
│   a. 代码变更说明 — 本次改了什么，为什么           │
│   b. 循环依赖/架构违规检测                        │
│   c. 跨层调用缺陷分析（事务边界、循环调用等）      │
│   d. 影响范围分析（AI 验证 + 补充）               │
│   e. 潜在代码问题（空指针、并发、性能等）          │
│   f. 测试建议（针对本次变更的测试重点）            │
│   g. 审查结论                                    │
└─────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────┐
│ Step 3: 解析 LLM 响应 → CodeReviewResult          │
│   - 定义审查结果的数据模型                        │
│   - 解析 LLM 的结构化输出（JSON 或 Markdown）     │
│   - 验证结果的完整性                              │
└─────────────────────────────────────────────────┘
 ↓
┌─────────────────────────────────────────────────┐
│ Step 4: 渲染审查文档                              │
│   - MarkdownGenerator (复用已有 Markdown 渲染)    │
│   - HtmlGenerator (复用已有 HTML 渲染)           │
│   - 最终输出: 审查文档                            │
└─────────────────────────────────────────────────┘
```

### 4.3 AI Prompt 设计（核心）

```
你是一个资深的 Java 代码审查专家。请审查以下代码变更。

## 项目信息
- 项目：${projectName}
- 版本：${projectVersion}
- 分支：${branch}

## 变更文件及 Diff（核心审查对象）
${fileDiffsWithLineNumbers}

## 依赖关系图谱摘要
变更文件的依赖关系：
${dependencyGraphSummary}

## 影响范围分析
${impactScope}

请基于以上代码变更内容，输出结构化审查结果，严格按以下 7 个维度：

### 1. 代码变更说明
总结本次变更的核心目的和主要内容。每个变更文件简述做了什么、为什么。

### 2. 架构与依赖分析
- 是否存在循环依赖或依赖反向（下层依赖上层）？
- 是否存在架构分层违规（如 Controller 直接调 Repository）？
- 模块边界是否被破坏？

### 3. 潜在代码缺陷
逐文件审查，重点关注：
- 空指针风险
- 事务边界（@Transactional 使用是否正确）
- 并发安全问题
- 资源未释放
- API 兼容性问题
- 异常处理是否合理

### 4. 变更影响范围
基于代码变更分析，评估：
- 哪些模块/服务可能被间接影响
- 是否为向后兼容的变更
- 是否需要同步更新文档/配置

### 5. 测试建议
针对本次变更：
- 哪些逻辑路径最需要测试覆盖
- 边界条件和异常场景
- 是否需要集成测试或契约测试

### 6. 代码规范与风格（可选）
- 是否违反项目编码规范
- 命名是否合理
- 注释是否充分

### 7. 审查结论
**结论**: [通过/需修改/拒绝]
**风险等级**: [低/中/高]
**关键发现**: 列出必须修改的问题（如有）
**建议**: 整体建议
```

### 4.4 LLM 调用复用现有能力

`AiClassifier` 中已有：
- OpenAI 兼容接口调用（DeepSeek 等）
- Anthropic API 调用
- API key 管理（加密存储 + 解密）
- Provider/Model 配置

新增 `CodeReviewAiService` 复用 `AiClassifier` 的 `callLlmApi(prompt, maxTokens)` 方法，通过组合而非继承：

```java
@Component
public class CodeReviewAiService {
    private final AiClassifier aiClassifier;  // 复用已有 LLM 调用能力

    public CodeReviewResult review(CodeReviewContext context) {
        String prompt = buildReviewPrompt(context);
        String response = aiClassifier.callLlmApi(prompt, 4096);  // 复用
        return parseReviewResponse(response);
    }
}
```

> 注意：需要将 `AiClassifier.callLlmApi(prompt, maxTokens)` 的访问权限从 `private` 改为 `protected` 或 `public`，或提取出一个独立的 `LlmClient` 服务。

### 5.4 编排器扩展

`GenerationOrchestrator.generateSync()` 方法在现有文档生成完成后增加后处理：

```java
public DocumentResult generateSync(DocumentRequest request) {
    // ... 现有逻辑：获取 commits、去重、分类、生成发布文档 ...
    DocumentResult result = documentGenerator.generate(request);

    // 新增：如果勾选了代码审查，额外生成审查文档
    if (request.isUseCodeReview()) {
        try {
            // 1. 获取变更的源码和 diff（通过 JGit 或 GitLab API）
            CodeReviewContext context = codeReviewDataCollector.collect(
                request.getProjectId(), request.getBranch(),
                request.getSinceHash(), request.getUntilHash(),
                commits  // 已有 commit 列表，用于确定变更范围
            );

            // 2. 运行 code-review-graph 引擎
            CodeReviewGraph graph = graphEngine.analyze(context);

            // 3. AI 深度审查
            CodeReviewResult reviewResult = codeReviewAiService.review(
                CodeReviewContext.builder()
                    .project(request.getProjectName())
                    .version(request.getProjectVersion())
                    .branch(request.getBranch())
                    .commits(commits)
                    .fileDiffs(context.getFileDiffs())
                    .graph(graph)
                    .impactScope(graph.getImpactScope())
                    .build()
            );

            // 4. 生成审查文档
            String reviewContent = reviewReportGenerator.generate(
                reviewResult, request.getReviewFormat());
            result.setReviewContent(reviewContent);

            // 5. 保存文件
            String reviewFilePath = saveReviewFile(taskId, reviewContent, request.getReviewFormat());
            result.setReviewOutputPath(reviewFilePath);

        } catch (Exception e) {
            log.error("Code review failed, but document generation succeeded: {}", e.getMessage());
            // 审查失败不应影响发布文档的生成结果
        }
    }

    return result;
}
```

> 审查失败不应阻塞发布文档的正常生成和返回。

## 8. 实施路线

### 第一阶段（MVP）
1. 抽取独立的 `LlmClient` 服务（从 `AiClassifier` 中提取 LLM 调用层）
2. 实现 `CodeReviewDataCollector` — 通过 JGit 获取变更文件 diff
3. 实现 `GraphBuilder` — 文件级 import 依赖图（正则解析）
4. 实现 `CodeReviewAiService` — 组装 prompt + 调用 LLM + 解析结果
5. 实现 `ReviewReportGenerator` — Markdown 输出
6. 集成到 `GenerationOrchestrator`
7. UI/API 变更

### 第二阶段
1. 方法级调用链分析
2. 引入 JavaParser 做 AST 解析（精度提升）
3. HTML 模板渲染 + 图可视化（Mermaid）

### 第三阶段
1. 增量审查（只审查未审查过的变更）
2. 审查规则可配置化
3. 审查结果持久化到数据库
4. 审查统计面板

## 11. 核心时序

```
用户/CI → 提交生成请求 (useCodeReview=true)
  │
  ▼
GenerationOrchestrator.generateSync()
  │
  ├── 获取 commits（已有逻辑）
  ├── 去重 + 分类（已有逻辑）
  ├── 生成发布文档（已有逻辑）
  │
  ├── (可选) CodeReviewDataCollector.collect()
  │     ├── JGit: git diff sinceHash..untilHash
  │     └── 输出: List<FileDiff> + 变更文件源码
  │
  ├── CodeReviewGraphEngine.analyze()
  │     ├── JavaFileParser 解析每个变更文件
  │     └── GraphBuilder 构建依赖图
  │     └── ImpactAnalyzer 分析影响范围
  │
  ├── CodeReviewAiService.review()
  │     ├── 组装 prompt（diff + 图谱 + 影响范围）
  │     ├── LlmClient.call(prompt)
  │     └── 解析响应 → CodeReviewResult
  │
  └── ReviewReportGenerator.generate()
        └── 输出审查文档 (MD/HTML)
  │
  ▼
返回 DocumentResult（含发布文档 + 审查文档路径）
```
