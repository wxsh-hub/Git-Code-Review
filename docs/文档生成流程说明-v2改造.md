# 文档生成流程说明 — 改造后 v2

## 整体流程

```
用户提交请求
  │
  ├─ 1. 拉取 Commit（GitLab API）
  │
  ├─ 2. AI 提交分类（可选，效率分析的前置条件）
  │
  ├─ 3. 生成基础报告（更新日志 + 贡献者分析）
  │     └─ MarkdownGenerator + DocumentGenerator
  │
  ├─ 4. AI 智能代码审查（可选）  ←── 本次改造核心
  │     │
  │     ├─ 4.1 CodeReviewDataCollector 拉取仓库代码
  │     ├─ 4.2 CodeReviewGraphEngine 静态架构分析
  │     ├─ 4.3 CodeReviewAiService 三步审查流水线
  │     │     ├─ Step 1: 文件预筛选（跳过 DELETE/非代码文件）
  │     │     ├─ Step 2: 按业务链路分组（↓ ModulePathResolver）
  │     │     └─ Step 3: 逐组调用 OCR MCP server
  │     │
  │     ├─ 4.4 审查后处理管线 (PostProcessor Pipeline)
  │     │     ├─ Phase 5: FindingBlameTracer   — git blame 追溯 P0/P1 引入者
  │     │     ├─ Phase 6: ReviewLlmService     — 双 LLM 交叉验证
  │     │     ├─ Phase 3: SecretDetector       — 敏感信息脱敏
  │     │     └─ Phase 3: FindingVerifier      — 行号/去重/误报/完整性校验
  │     │
  │     └─ 4.5 ReviewReportGenerator 生成四页报告
  │           ├─ 第一页：管理摘要
  │           ├─ 第二页：问题处置页
  │           ├─ 第三页：模块与趋势页 (ModuleReportGenerator)
  │           └─ （第四页由效率分析补充）
  │
  ├─ 5. 开发者效率分析（可选）  ←── 本次改造扩展
  │     ├─ 5.1 DeveloperEfficiencyService: fix commit blame 聚合
  │     ├─ 5.2 跨管线数据共享: CodeReviewResult.findings → EfficiencyAnalysisResult
  │     └─ 5.3 EfficiencyReportGenerator 生成
  │           ├─ 开发者 Bug 详情（含置信度/状态列）
  │           ├─ 未修复漏洞详情（7.6 跨管线）
  │           ├─ 开发者综合画像（已修复+未修复双维度）
  │           ├─ 协作信号（克制表述，不做人格推断）
  │           └─ 第四页并入审查报告末尾
  │
  └─ 6. 合并输出 ← 旧版串行拼接；新版的审查+效率内容已内聚
```

## 改造前 vs 改造后

### 报告输出结构

| | 改造前 | 改造后 |
|---|---|---|
| **更新日志** | 在最前面，一个文件串到底 | 保留在基础报告中 |
| **代码审查报告** | 段落式 7 维 LLM 输出，按文件分组 | **四页分层结构**（管理摘要→问题处置→模块趋势→效率附录） |
| **效率分析报告** | 关键观察（含人格推断）+ 简单表 | 置信度/状态列 + 未修复漏洞详情 + 综合画像 + 协作信号 |
| **贡献者统计** | 按天聚合，硬编码 10% 低频阈值 | **按月聚合**，相对公式低频阈值 |
| **报告受众** | 所有人看同一份 | **四页四种受众**：高管/经理/总监/管理者 |

### 代码审查

| | 改造前 | 改造后 |
|---|---|---|
| **审查方式** | 全量 diff 一次灌入 LLM prompt | **三步流水线**：预筛选→业务链路分组→逐组 OCR |
| **问题模型** | OcrComment / CodeReviewResult 各自为政 | **统一 Finding**，OCR/Legacy/Bug 归因/Secret 四种来源收敛 |
| **安全** | 无脱敏 | Phase 1 8 种基础脱敏 + Phase 3 11 条增强规则 |
| **质量把关** | LLM 输出直接进报告 | **四道校验**：行号/去重/误报检测/触发完整性 |
| **漏洞追溯** | 无 | **git blame** P0/P1 漏洞，三级回退 |
| **双 LLM 验证** | 单 LLM 直接判定 | **交叉验证**：置信度平均 ≥0.7 才确认 |
| **模块聚合** | 无 | ModulePathResolver 提取领域 + 风险分排名 + 重复问题类型 |

### 效率分析

| | 改造前 | 改造后 |
|---|---|---|
| **表述风格** | "质量意识有待提高""修复最多" | **只做事实陈述**："X 的 commit 被后续修复关联（已确认 Y 个）" |
| **数据说明** | 无 | **固定注释块**：git blame ≠ bug 引入，低贡献率 ≠ 不努力 |
| **未修复漏洞** | 无 | 代码审查发现同步到效率报告，**修复+未修复双维度画像** |
| **置信度/状态** | 无 | Bug 表新增置信度 % 和已确认/误报状态列 |

### 关键技术提升

| | 改造前 | 改造后 |
|---|---|---|
| **OCR 集成** | 单次 LLM prompt | stdin/stdout JSON-RPC MCP 子进程 |
| **数据模型** | 各模块独立定义 | Finding 统一模型 + PostProcessor 可扩展管线 |
| **报告链接** | 无 | codeLink 自动拼接（GitHub/Gitee /blob/, GitLab /-/blob/） |
| **截止时间** | 无 | P0=立即, P1=+3 工作日, P2=+7 工作日, 跳过周末 |
| **审查范围** | 口径混乱 | ReviewScope 枚举 + 报告第一行标明 |
