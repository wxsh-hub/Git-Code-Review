## 1. 产品概述

### 1.1 简介

devops-ai 是一款通用的项目发布文档自动生成工具。它从 GitLab 仓库获取提交记录（Commits），对提交进行分类整理，并生成结构化的发布文档（CHANGELOG / Release Notes），支持 Markdown 和 HTML 两种输出格式。

### 1.2 核心功能

| 功能 | 说明 |
|------|------|
| GitLab 集成 | 支持 API 和 Git 克隆两种模式连接 GitLab 仓库 |
| 提交分类 | 自动将提交归类为 新功能、Bug修复、代码重构 等 12 个类别 |
| AI 智能分析 | 接入 LLM（DeepSeek / OpenAI / Anthropic）进行智能分类和分析报告生成 |
| 多模板支持 | 内置标准、简洁、详细三种模板，支持自定义模板 |
| 增量生成 | 仅处理上次生成后的新提交，避免重复处理 |
| 代码审查 | 基于 AI 的代码差异审查，生成结构化审查报告 |
| 多种输出 | 支持 Markdown 和 HTML 格式输出文档 |
| REST API | 提供完整的 HTTP API 供第三方系统对接 |

## 2. 快速开始

### 2.1 启动系统

系统提供两种运行方式：前台模式（调试推荐）和后台模式（服务器推荐）。

#### 方式一：启动脚本（推荐）

```bash
# ── Windows ──
start.bat                  # 前台运行，Ctrl+C 或关窗停止
start.bat --daemon         # 后台运行（任务管理器停止）

# ── Linux / Mac ──
./start.sh                 # 前台运行（默认），Ctrl+C 停止
./start.sh start           # 后台启动
./start.sh stop            # 停止后台服务
./start.sh status          # 查看运行状态
./start.sh restart         # 重启
```

启动成功后访问：**http://localhost:8070**

> **后台管理**：Linux 脚本使用 PID 文件（`.app.pid`）管理进程，`stop` 命令会优雅关闭服务, 默认端口为 8070，如需修改，编辑 `application.yml` 中的 `server.port`

### 2.2 首次使用向导

首次启动时，系统会自动跳转到初始化向导页面，您需要：

1. **填写配置名称**（如"公司 GitLab"）
2. **选择连接方式**：
   - **克隆模式（推荐）**：适用于 GitLab CE v18.x，使用 Git 协议克隆仓库，填写完整克隆地址、用户名和密码
   - **API 模式**：使用 GitLab API，需 Personal Access Token
3. 点击"测试连接并保存"

配置完成后，系统将跳转至项目配置页面。

### 2.3 完成一次文档生成

四个步骤即可完成文档生成：

**步骤 1**：在「项目配置」页面添加 GitLab 项目和配置
**步骤 2**：在「AI 模型配置」页面配置 LLM API 密钥（可选，但推荐）
**步骤 3**：进入「文档生成」页面，选择项目配置和参数
**步骤 4**：点击「开始生成」，完成后在「生成历史」页面查看结果

## 3. 系统配置

### 3.1 项目配置

**路径**：侧边栏 → 项目配置 | `/config/project`

#### 新增项目

| 字段 | 说明 | 必填 |
|------|------|------|
| 项目名称 | 自定义的项目显示名称 | 是 |
| 项目编码 | 自动生成，可手动修改，第三方 API 调用时使用此编码 | 是 |
| GitLab 地址 | API 模式下为 GitLab 根地址；克隆模式下为完整 Git 仓库地址 | 是 |
| 认证方式 | Token 认证 或 用户名+密码 | 是 |
| Personal Access Token | Token 认证时填写，需勾选 `read_api` 权限 | 按认证方式 |
| 用户名 / 密码 | 密码认证时填写 | 按认证方式 |

填写完连接信息后，点击「测试连接」验证连通性。连接成功后：

- 选择 GitLab 项目（自动拉取项目列表）
- 选择默认分支
- 选择文档模板

#### 已有项目管理

项目列表显示所有已配置的项目，可查看每个项目的编码、GitLab 地址、默认分支和模板，支持删除操作。

### 3.2 AI 模型配置

**路径**：侧边栏 → AI 模型配置 | `/config/ai`

配置 LLM 接口用于提交消息的智能分类和分析报告生成。

| 字段 | 说明 | 默认值 |
|------|------|--------|
| 模型供应商 | DeepSeek / OpenAI / Anthropic | DeepSeek |
| 模型 ID | 使用的模型名称 | deepseek-chat |
| 自定义请求地址 | API 请求地址，留空使用供应商默认地址 | 自动 |
| API 密钥 | LLM 服务的 API Key，加密存储 | — |
| 温度参数 | 0.1-2.0，分类建议 0.1-0.3 | 0.3 |

**支持接入的 LLM**：

- DeepSeek：`https://api.deepseek.com`，模型 `deepseek-chat`
- OpenAI：`https://api.openai.com`，模型 `gpt-4o-mini`
- Anthropic：`https://api.anthropic.com`，模型 `claude-3-haiku`
- 任意兼容 OpenAI API 格式的自定义接口

配置完成后，点击「测试连接」验证 API 连通性。

> **提示**：未配置 AI 时，系统将自动使用规则分类作为后备方案，仍可正常生成文档。

## 4. 文档生成

### 4.1 生成页面

**路径**：侧边栏 → 文档生成 | `/generate`

#### 参数说明

| 参数 | 说明                                                                        | 必填 |
|------|---------------------------------------------------------------------------|----|
| 项目配置 | 选择已配置的 GitLab 项目                                                          | 是  |
| 分支 | 选择要生成文档的分支                                                                | 是  |
| 项目版本 | 版本号，如 v1.0.0                                                              | 推荐 |
| 筛选方式 | 时间范围 或 Hash 范围（二选一）                                                       | 可选 |
| 起始时间 | 从指定时间开始获取提交                                                               | 可选 |
| 截止时间 | 到指定时间为止获取提交，默认当前时间                                                        | 可选 |
| 起始提交 Hash | 从指定 Commit 开始                                                             | 可选 |
| 截止提交 Hash | 到指定 Commit 为止                                                             | 可选 |
| 作者筛选 | 只包含指定作者的提交                                                                | 可选 |
| 输出格式 | Markdown 或 HTML                                                           | 否  |
| 文档模板 | standard(标准发布文档含项目名称、版本、构建信息、变更分类) / compact(简洁) / detailed(详细模式，含构建信息、分类详情、提交统计) | 是  |
| 分组维度 | 提交的分组方式（多选）                                                               | 可选 |
| 增量生成 | 只处理新增的提交                                                                  | 可选 |
| AI 智能分析 | 启用 LLM 进行分类和分析报告                                                          | 可选 |

### 4.2 提交分类体系

系统将提交自动分为以下 12 个类别：

| 类别 | 说明 |
|------|------|
| 新功能 | 新增的功能特性 |
| 功能更新 | 对现有功能的修改和更新 |
| Bug修复 | 缺陷修复和问题解决 |
| 代码重构 | 代码结构调整，功能不变 |
| 性能优化 | 性能改进和效率提升 |
| 样式调整 | UI 和样式变更 |
| 文档更新 | 文档和注释变更 |
| 测试相关 | 测试用例和测试框架变更 |
| 构建相关 | 构建脚本和配置变更 |
| CI配置 | CI/CD 流水线变更 |
| 依赖更新 | 依赖库版本更新 |
| 其他变更 | 无法归入以上类别的变更 |

#### 分类模式

- **规则分类（默认）**：根据提交消息前缀关键字匹配（如 `feat:`、`fix:`、`refactor:` 等）
- **AI 分类（启用 AI 智能分析）**：调用 LLM 理解提交消息语义进行分类，准确度更高

### 4.3 AI 分析报告

当启用「AI 智能分析」且选择「详细」模板时，系统会自动生成一份版本分析报告，包含：

1. **开发者贡献分布** — 按提交次数排序的开发者贡献量统计
2. **提交类型统计** — 各类别提交的数量和占比
3. **版本核心价值** — 本次迭代的核心目标与业务价值
4. **核心开发成果总结** — 重要功能、技术优化和关键突破

### 4.4 同步模式

API 接口支持同步模式（`sync=true`），提交任务后轮询等待完成并直接返回文件内容，适用于需要在 CI/CD 流水线中集成使用的场景。

## 5. 模板管理

### 5.1 内置模板

| 模板名称 | 显示名称 | 说明 |
|----------|----------|------|
| standard | 标准模板 | 包含项目名称、版本、构建信息、变更分类 |
| compact | 简洁模板 | 仅包含项目名称和变更内容，适合快速预览 |
| detailed | 详细模板 | 包含构建信息、统计数据和变更分类详情 |

### 5.2 模板变量

模板使用 `${variableName}` 语法进行变量替换，支持以下变量：

| 变量 | 类型 | 说明 |
|------|------|------|
| `${projectName}` | String | 项目名称 |
| `${projectVersion}` | String | 项目版本号 |
| `${branch}` | String | 分支名称 |
| `${buildTime}` | String | 构建/生成时间 |
| `${dateRange}` | String | 时间范围 |
| `${categories}` | String | 分类内容（自动渲染） |
| `${totalCommits}` | int | 提交总数 |
| `${totalAuthors}` | int | 参与开发者数 |
| `${analysisReport}` | String | AI 分析报告（仅在详细模板+启用AI时生效） |

### 5.3 自定义模板

在「模板管理」页面可新增、修改和删除自定义模板。

**操作步骤**：
1. 点击「新增自定义模板」
2. 填写模板名称、显示名称和描述
3. 在模板内容区域使用上述变量编写模板
4. 点击「预览模板」查看渲染效果
5. 点击「保存」

内置模板支持「恢复默认」操作，自定义模板支持删除。

## 6. 生成历史

### 6.1 查看历史

**路径**：侧边栏 → 生成历史 | `/history`

历史页面显示所有文档生成任务的记录，包含以下信息：

| 字段 | 说明 |
|------|------|
| 任务 ID | 唯一标识，格式：`项目编码_日期-时间` |
| 项目 | 关联的项目名称 |
| 状态 | pending（等待）/ running（运行中）/ completed（完成）/ failed（失败） |
| 格式 | 输出文档的格式 |
| 增量 | 是否为增量生成 |
| 审查 | 是否包含代码审查文档 |
| 创建时间 | 任务提交时间 |
| 完成时间 | 任务完成时间 |

### 6.2 查看和下载

已完成的任务支持以下操作：

- **查看文档**：在浏览器中直接查看生成的文档
- **下载文档**：下载生成的文档文件
- **删除**：删除历史记录及对应的输出文件

### 6.3 任务状态说明

| 状态 | 说明 | 可操作 |
|------|------|--------|
| pending | 任务已提交，等待执行 | — |
| running | 任务正在执行中 | — |
| completed | 任务已完成，文档已生成 | 查看 / 下载 / 删除 |
| failed | 任务执行失败 | 查看错误信息 / 删除 |

## 7. 第三方对接 API

**基础地址**: `https://<your-host>/api/v1`
**Content-Type**: `application/json`

完整 API 文档也可通过 Swagger UI 访问：`http://localhost:8070/swagger-ui.html`

### 7.1 提交文档生成任务

#### GET /api/v1/generate（快速提交）

查询参数方式提交，支持同步/异步两种模式。

**查询参数**:

| 参数 | 必填 | 说明 | 默认值 |
|------|------|------|--------|
| `projectCode` | **是** | 项目编码 | - |
| `branch` | 否 | Git分支 | 项目默认分支 |
| `templateName` | 否 | 文档模板名 | 项目默认模板 |
| `format` | 否 | 输出格式 | `markdown` |
| `projectVersion` | 否 | 版本号 | - |
| `author` | 否 | 作者筛选 | - |
| `sinceHash` | 否 | 起始 commit hash | - |
| `incremental` | 否 | 增量模式 | `false` |
| `useAiClassifier` | 否 | AI 分类 | `false` |
| `sync` | 否 | 是否同步模式 | `false` |

**异步模式**（`sync=false`，默认）：返回 `taskId`。

```bash
curl "https://<your-host>/api/v1/generate?projectCode=user-service&branch=main&templateName=standard&format=markdown&projectVersion=v1.2.0"
```

**同步模式**（`sync=true`）：轮询等待最多 3600 秒，任务完成后直接返回文件下载，不返回 JSON。

```bash
curl "https://<your-host>/api/v1/generate?projectCode=user-service&branch=main&templateName=standard&format=markdown&projectVersion=v1.2.0&sync=true"
```

### 7.2 查询任务状态

#### GET /api/v1/tasks/{taskId}

```bash
curl "https://<your-host>/api/v1/tasks/user-service_20260504-143022"
```

**响应示例**:

```json
{
  "success": true,
  "data": {
    "taskId": "user-service_20260504-143022",
    "status": "completed",
    "progress": 100,
    "createdAt": "2026-05-04T10:00:00",
    "completedAt": "2026-05-04T10:00:05",
    "format": "markdown",
    "outputPath": "/output/.../CHANGELOG.md",
    "errorMessage": null
  }
}
```

**status 说明**:

| 状态 | 说明 |
|------|------|
| `pending` | 排队等待处理 |
| `running` | 正在生成文档 |
| `completed` | 生成成功，可下载文件 |
| `failed` | 生成失败，查看 `errorMessage` |

### 7.3 获取生成结果

#### GET /api/v1/tasks/{taskId}/download（推荐）

以文件流方式下载生成的文档，响应头包含 `Content-Disposition: attachment`。

- 格式为 `markdown` 时，Content-Type: `text/markdown`
- 格式为 `html` 时，Content-Type: `text/html`

```bash
curl "https://<your-host>/api/v1/tasks/user-service_20260504-143022/download"
```

#### GET /api/v1/tasks/{taskId}/result

以文本方式返回生成的文档内容，可直接在页面展示或二次处理。任务未完成时返回 HTTP 409。

```bash
curl "https://<your-host>/api/v1/tasks/user-service_20260504-143022/result"
```

### 7.4 建议对接流程

#### 异步模式（推荐）

```
① GET /api/v1/generate?projectCode=xxx&...        → 提交任务，获取 taskId
② GET /api/v1/tasks/{taskId}                       → 每 2-3 秒轮询状态直到 completed
③ GET /api/v1/tasks/{taskId}/download              → 下载最终文档
```

#### 同步模式（适合快速调用）

```
① GET /api/v1/generate?projectCode=xxx&sync=true   → 等待最多 120 秒，直接返回文件
```

### 7.5 错误码

| HTTP 状态码 | 说明 |
|-------------|------|
| 200 | 成功 |
| 400 | 参数错误（projectCode 为空、时间与 hash 范围冲突、项目编码不存在等） |
| 404 | 任务不存在 |
| 408 | 同步模式超时（超过 120 秒） |
| 409 | 任务尚未完成（调用了 result/download 但任务仍在运行） |
| 500 | 服务端错误 |

**错误响应示例**:

```json
{
  "success": false,
  "message": "projectCode不能为空，必须指定项目编码",
  "data": null
}
```

### 7.6 Jenkins Pipeline 对接示例

```groovy
pipeline {
    environment {
    }
    stages {
        stage('Generate Release Notes') {
            steps {
                script {
                   try {
                      def fileName = "CHANGELOG-${projectCode}-${PROJECT_VERSION}.md"
                      def apiUrl = "http://<your-host>:8070/api/v1/generate?projectCode=${projectCode}&branch=${git_version}&format=markdown&templateName=detailed&projectVersion=${PROJECT_VERSION}&incremental=false&sync=true&useAiClassifier=true"

                      sh "curl -s -o ${fileName} --max-time 1800 \"${apiUrl}\""

                      def fileSize = sh(returnStdout: true, script: "wc -c < ${fileName}").trim()
                      echo "更新日志已生成: ${fileName} (${fileSize} bytes)"
                      archiveArtifacts(
                              artifacts: "${fileName}",
                              fingerprint: true,
                              allowEmptyArchive: false,
                              onlyIfSuccessful: true,
                              defaultExcludes: false
                      )
                   } catch (Exception e) {
                      echo "警告: ${projectCode} 更新日志生成失败: ${e.message}"
                   }
                }
            }
        }
    }
}
```

## 8. 常见问题

### 8.1 连接 GitLab 失败

**可能原因**：
- 克隆模式下使用了 HTTPS 地址但端口不正确
- API 模式下 Token 权限不足，需至少勾选 `read_api`
- GitLab CE v18.x 默认禁用密码 API 认证，请使用克隆模式或 Token 认证

**解决方案**：
- 克隆模式建议使用完整的 Git 仓库地址
- API Token 需要在 GitLab Settings → Access Tokens 中创建

### 8.2 AI 分类不生效

**确认以下配置**：
- AI 模型配置中的 API Key 已正确填写
- 点击「测试连接」返回成功
- 在生成文档时勾选了「AI 智能分析」

如果 AI 不可用，系统会自动降级为规则分类，不影响文档生成。

### 8.3 生成的文档为空

**可能原因**：
- 筛选的时间范围或 Hash 范围内没有提交
- 分支名称不正确
- 项目尚未有任何提交

**排查方法**：检查生成历史中的任务状态，如果显示失败可查看错误信息。

### 8.4 如何修改输出目录

编辑 `application.yml` 中的 `devops.ai.generator.output-dir` 配置项，默认为 `./output`。

### 8.5 CI/CD 集成建议

- 使用同步模式（`sync=true`）获取文件直接输出
- 使用项目编码（projectCode）而不是数据库 ID 进行调用
- 定期使用 AI 分类生成详细报告，日常构建使用增量生成
- 使用 `useCodeReview=true` 开启代码审查功能

## 附录

### A. 配置文件参考

```yaml
server:
  port: 8070

spring:
  datasource:
    url: jdbc:h2:file:./data/devops-ai;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE
    driver-class-name: org.h2.Driver
  jpa:
    hibernate:
      ddl-auto: update

devops:
  ai:
    encrypt:
      key: devops-ai-secret-key-2026
    generator:
      output-dir: ./output
      default-template: standard
      default-format: markdown
    cache:
      project-ttl-minutes: 60
      branch-ttl-minutes: 5
    task:
      max-concurrent: 5
```

### B. 输出目录结构

```
output/
├── user-service_20260504-143022.md              # 生成的 Markdown 文档
├── user-service_20260504-143022.html            # 生成的 HTML 文档
```

### C. 数据库说明

系统默认使用 H2 文件数据库（`./data/devops-ai.mv.db`），生产环境建议切换为 MySQL：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/devops_ai?useSSL=false&characterEncoding=utf-8
    driver-class-name: com.mysql.cj.jdbc.Driver
    username: root
    password: your-password
```
